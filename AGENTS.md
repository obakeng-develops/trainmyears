## Guides

### Cognitive Load
- Cognitive Load Is What Matters — explains how extraneous complexity slows teams and offers practical heuristics for keeping code and architecture easy to understand. https://github.com/zakirullin/cognitive-load

### API Design
- Everything I Know About Good API Design — argues for boring, stable APIs with careful change management, and covers versioning, idempotency, pagination, and rate limits. https://www.seangoedecke.com/good-api-design/

### Events and Observability
- A Practitioner's Guide to Wide Events — shows how to instrument systems with one rich event per request and how to query it effectively for debugging. https://jeremymorrell.dev/blog/a-practitioners-guide-to-wide-events/

### Logging
- Logging Sucks — makes the case for wide, high-cardinality structured events instead of scattered log lines, with pragmatic guidance on sampling and context. https://loggingsucks.com/

### ADRs
- How to Create Architectural Decision Records (ADRs) — and How Not To — outlines what makes ADRs useful, highlights common anti-patterns, and stresses clear tradeoffs and accountability. https://www.ozimmer.ch/practices/2023/04/03/ADRCreation.html
- How to Review Architectural Decision Records (ADRs) — and How Not To — provides review perspectives, tactics, and checklists for actionable, respectful ADR feedback. https://www.ozimmer.ch/practices/2023/04/05/ADRReview.html
- A Definition of Done for Architectural Decision Making — proposes evidence, criteria, agreement, documentation, and review planning as a checklist for “done” decisions. https://www.ozimmer.ch/practices/2020/05/22/ADDefinitionOfDone.html

### Code Review
- The Standard of Code Review — sets the bar for approving changes that improve code health while balancing progress and quality. https://google.github.io/eng-practices/review/reviewer/standard.html
- What to Look for in a Code Review — detailed checklist for design, correctness, tests, complexity, naming, docs, and overall code health. https://google.github.io/eng-practices/review/reviewer/looking-for.html
- Stacked Diffs — explains the stacked pull request workflow, why it helps with review throughput, and how to manage dependencies between changes. https://newsletter.pragmaticengineer.com/p/stacked-diffs

### Design Patterns
- Design Patterns (Refactoring.Guru) — clear catalog of classic patterns with intent, tradeoffs, and examples to build a shared vocabulary. https://refactoring.guru/design-patterns
